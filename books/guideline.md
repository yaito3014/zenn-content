# 執筆方針

この文書は、**C++23入門シリーズ全体**(基本編 + 続きの編)を執筆・編集する際の内部指針である。編をまたぐため、個々の本の中ではなく一段上(`books/` 直下)に置く。公理・定理・規律は全編に適用する。章立ては編ごとに分け、現状は基本編のみ確定している。以下で「本書」とあるのは、特に断りがなければ基本編を指す。

読者に向けて表明する文書ではなく、判断に迷ったときに立ち返るための規律として用いる。スタンスは読者に宣言するのではなく、各章の本文で一貫して体現することで伝える。

本書は規格を公理系として捉える。この文書もまた、その精神にならい、少数の**公理**を先に置き、個々の判断を公理から導かれる**定理**として並べる。定理には、それがどの公理から導かれるかを註記する。複数の公理に依拠する定理もある。

---

## 公理

### 公理1：正確さ

本書は、C++ 規格を数学における公理系のように捉える。C++ の振る舞いはすべて規格に遡って正当化され、規格は C++ について語るときの出発点である。本書の役割は、この公理系を読み手にインストールする助けとなることであり、公理そのものが「わかりやすさ」のために歪められてはならない。歪んだ公理を教えれば、読者がそこから導く帰結はすべて歪み、推論の土台が汚染されるためである。

これは同時に、本書の記述が**反証可能**であることを要請する。規格という公理に照らせば、本書の記述の正誤は判定できる。本書は、反証可能な記述だけで構成する。

公理系のインストール(既存の入門書との差別化の核でもある)と反証可能性は、別々の理由ではなく1つの思想の両面である。規格を公理系として据えるからこそ本書の記述は反証可能になり、反証可能であろうとするからこそ公理を歪めずにインストールする必要が生じる。

### 公理2：広さ

本書はなるべく広い層に届ける。読者層を不必要に狭めない。

### 優先規則：公理1 が公理2 に優先する

2つの公理はときに緊張する。正確さを徹底すると硬く難しくなり、広い層には届きにくくなる。広さを追えば、正確さを犠牲にして易しくしたくなる。

衝突した場合、公理1(正確さ)が上位にある。広さのために公理を歪めることはしない。

ただし衝突を「どちらかを諦める」形では解かない。正確さを保ったまま、広さを最大化する手段を探す。これが多くの定理の形を決めている。

## 定理

### 文体・表現

**規格が定めていることは、ためらわず断定する。**(公理1)
曖昧にせず言い切る。「言い切りを避ける」こと自体が美徳なのではない。正確なことは、はっきり断定する。
- 定義は宣言の一種である
- 文(statement)はトップレベルには現れず、関数定義のような宣言の本体に書かれる
- 宣言の並び(`declaration-seq`)の順序には意味があり、名前は原則として使用箇所より前で宣言される
- 仮引数は `parameter-declaration` であり、文法カテゴリ `declaration` とは別である
- プリプロセッシングディレクティブは宣言ではない

**正確でないことは、断定を避ける。**(公理1)
断定を避けるべきなのは、不正確になる場合に限られる。2つの型がある。
- *型1：規格上そう言い切れないこと。* 近似であることを示す。例：「宣言は名前を導入する」は `static_assert` 等で反証されるため、「多くの宣言は名前を導入する」と書く。例：declaration を一言の定義として書こうとしない。規格 [basic.pre] が与えているのは「A declaration may (re)introduce one or more names and/or entities into a translation unit. If so, the declaration specifies the interpretation and semantic properties of these names.」という条件付き(may / if so)の効果記述であって、初学者の本文にそのまま置ける一言定義の形ではない。本書は無理に定義せず、観察(「このコードには宣言が3つある」)から入る。
- *型2：検証しようのない主観的評価(効果・優劣)。* 「役に立つ」「理解が進む」「こう書くべき」とは言わず、「本書はこの見方を主軸に据える」とスタンスの表明にとどめる。押し付けを避け、読者が採否を判断する余地を残す。

**異なるカテゴリーを語のレベルで区別する。**(公理1)
形式と意味、名前と構造、型と値、抽象と具体など、混同しがちなカテゴリーを語の選び方で分ける。精度を保つ最も実践的な規律であり、推敲で出てくる指摘の多くがここに集約される。
- 式と値を区別する：「`return` のあとに書いた値」ではなく「`return` のあとに書いた式の値」。
- 名前と構造を区別する：「`add` は宣言」ではなく「以下に示すのは宣言」。実体と構造の場合も同様(「関数は宣言の一種」ではなく「関数の宣言は宣言の一種」)。
- 型と値を区別する：「`int` どうしの和」ではなく「`int` 型の値どうしの和」。
- 抽象と具体を区別する：「C++ を読み解く」のように何を読むか不明瞭な表現を避ける。
- 規格と内部資料(このガイドライン等)も別出典。内部資料の記述を鵜呑みにせず原文に当たる。
- 一箇所で区別を直したら、同系統がファイル全体に残っていないか sweep する。

**擬人化を避ける。**(公理1)
「コンパイラに教える」という擬人化は、宣言の効果を「コンパイラという主体への働きかけ」として描くが、規格は宣言の効果を「プログラムに対する効果」として定めている。規格にない主体(コンパイラの意思や理解)を持ち込んで公理の像を歪めるため、避ける。
- 宣言を「C++ に教える」ではなく「プログラムに導入する」とする。

**常体(である調)で通す。**(公理1)
究極的には書き手の好み。理由を述べるなら、です・ます調がもたらす「読み手への敬意」は、公理系を読み手にインストールするという目的には必要のないものだからである。敬体でも公理は導入できるが、敬体が乗せる関係性の情報は本書の目的にとって余剰である。公理の伝達に不要なものを乗せない、という現れ。

**励まし・難易度評価・ペースの許可に慎重である。**(公理1)
禁じるというより、多くの場合ノイズになる。
- 励まし(「がんばろう」等)は読者の状態についての書き手の推測にすぎず、励まされたいと思っていない読者にはノイズになる。
- 「これは難しい」という難易度評価は主観であり、書き手のポジショントークになりやすく、読者の感じ方を先回りして決めつける。検証不能な主観として公理1(反証可能性)に反する。
- 「立ち止まってよい」というペースの許可は、文章において読者がもともと握っている自由を本が与えるかのように振る舞うもので、不要。

**「コンパイル」より「ビルド」を選ぶ。**(公理2)
公理の正確さではなく、読者の負荷を下げる配慮による。コンパイルという翻訳の段階の話を序盤で背負わせず、「ビルドツール」という素朴な語にとどめ、読者がその時点で理解すべきことを減らす。両立しない場合は正確さを優先する(優先規則)。

**標準名は受動形、本書固有の名付けは「本書では」で導入する。**(公理1)
- 日本語圏で標準的な訳語(文・リテラル・プリプロセッシングディレクティブ等)には「本書では」を付けず、「と呼ばれる」のような受動形で導入する。
- 本書独自のフレーミングを名付けるときだけ「本書では〜と呼ぶ」と出典を明示する。
- 内部資料(このガイドライン等)でだけ通用する用語(例：アンカーコード)は本文に持ち込まない。読者にとって未定義の語を含めない。

**ダッシュ(―)を多用しない。**(公理系の外：媒体慣習)
これは公理から導かれない。Zenn をはじめとする和文のインターネット技術文書において、ダッシュは相対的に見慣れない記号であり、読者を一瞬立ち止まらせる。記号の珍しさで余計な負荷をかけない。和文では読点・括弧・文分割で代替する。媒体の慣習という統計的事実に基づくため、慣習が変われば見直しうる。

**和文中の括弧は半角 `()` を使う。**(公理系の外：表記)
和文の括弧は半角 `()` で揃える。全角 `（）` は使わない。鉤括弧「」は全角のまま使う。

**`:::details` の summary 行に markdown を書かない。**(公理系の外：媒体仕様)
Zenn では `:::details` と同じ行に書く summary(折りたたみの見出し)に markdown が反映されず、バッククォート等はそのまま記号として表示される。summary はプレーンテキストで書く(例：「`const` と…」ではなく「const と…」)。コラムの中身(`:::` で囲んだ本体)は通常どおり markdown が効くので、そちらは従来どおり。

**`const` は型の後ろに置く(east const)。**(公理1／公理2と緊張)
`const int` ではなく `int const` と書く。`const` はその左に書いたものを修飾する、と一貫して読めるため。ポインタ(`int const *` は指す先が const、`int * const` はポインタ自体が const)でも規則が崩れない。west const(`const int`)は先頭に限り前置を許す特例で、east const のほうが規格の一般規則に素直。少数派の表記で見慣れなさという公理2 側のコストはあるが、一貫性(公理1)を優先する。const を導入する章で、前置も同じ意味だが本書は後置に統一する旨を一度だけ断る。

### 読者と本文の設計

**三層の対象読者すべてに居場所を用意する。**(公理2)
(1) プログラミングを初めて学ぶ人、(2) 他の言語をすでに習得している人、(3) 以前に C++ を学んだことのある人。

**特定の言語の知識を読者に仮定しない。差異の説明には複数の言語を例示する。**(公理2)
眼目は、読者が特定の言語を知っている前提で書かないこと。「Rust ではこうだが」と、読者が Rust を知っている前提で書いた瞬間、Rust を知らない読者を一瞬置いていく。
一方、他言語との差異を語るとき、実在の言語を一切挙げないと像が結びにくい。そこで次の形をとる。(1) 語る性質は言語によらない一般的なものとする。(2) それを裏づける例として、複数の言語を挙げる。複数挙げるのは、1つだけだとその言語を知る読者向けの記述に見え、知らない読者を置いていくため。(3) 例の言語を知らない読者でも、一般的な性質の記述だけで筋が通るようにする。
例：「言語によっては整数があふれると自動的に多倍長になる(Python など)。C++ の `int` には範囲があり…」のように、一般論を主、言語名を従とする。
現状、他言語比較コラムは次に置く:declaration(文中心 vs 宣言中心の見方)・variable(可変性の既定/Scala・Rust)・expression-and-statement(式中心の言語/Scala・Rust・Ruby)・integer-type(`/` の意味とあふれ/Python・Ruby)・multiple-files(取り込みのしくみ/Python・Java)・enum(enum が何を持てるか/Rust の直和型・Java のオブジェクト)・class(データと操作の結びつけ方/C=データのみ関数なし・Rust=操作は impl ブロックに別置き。加えて C との部分的互換:データだけの構造体は同配置で相互運用できるが C++ 独自の機能で崩れる)。新章を足すときは、この分布と偏りを見て要否を判断する。

**本文は本当の初学者にとって素直であること。**(公理2)
コードを読める前提を本文に持ち込まない(例：「素朴に答えるなら手続きと言いたくなる」は読解を要求するため不可)。本文の足場は、読者が既に得ているものに限る(例：「実行すると 5 と出力される」と既に述べた事実)。

**厳密さと個別の親切はコラムへ逃がす。**(公理1・公理2／優先規則)
本文では公理2(誰でも読める素直さ)を保ち、公理1(厳密さ)はコラムで満たす。両立を諦めず、構造で解く本書の中心的な手段。コラムが引き受けるもの：文法的厳密さ(statement の包含、名前を導入しない宣言、`declaration-seq` 等)、他言語・旧 C++ との対比、経験者にこそ響く視点(「手続きとして捉える見方」の位置づけ等)。

**構成の透明性を保つ。**(公理1・公理2／優先規則)
いま何を扱い、何を後回しにし、なぜそうするのかを正直に示す。難しさを難易度評価(主観＝公理1違反)で語らず、扱う範囲と順序の明示に置き換える。これにより読者は現在地を見失わない。「とっつきにくさ」の正体は「いまどこにいて、この先どうなるか分からない」という見通しのなさにあり、温かい言葉ではなく見通しがそれを解く。
- 「`main` は特別な関数であり、その特別さについては関数の章で扱う」
- 範囲は事実として示す(「この章で扱うのは〇〇という点である」「後の章で扱う」)。読み手への許可の形(「〇〇だけ押さえておけばよい」「順に読めばよい」)は避け、何を扱い何を後回しにするかを事実で述べる(ペースの許可を避ける定理と同じ精神)。

**語を出すことと、定義することを区別する。**(公理1・公理2／優先規則)
種類のラベル(関数・変数)や特別な存在(`main`)は、名前を出しても中身は該当章へ送ってよい。理解の負荷を分散して公理2を保ちつつ、不正確な定義(公理1違反)を避ける。

### 色付きボックスの規律(Zenn message 機能)

**黄と赤を厳密に使い分ける。**(公理1)
- `:::message`(黄)：ビルドツールが指摘する誤り(ビルドが成功しない誤ったコード)。
- `:::message alert`(赤)：ビルドが成功してしまう、とりわけ実行時に深刻な問題(未定義動作など)を引き起こしうる誤り。

赤は実行時 UB 系に厳格に絞る。ビルドは通るが UB ではない単なるロジックの誤りには使わない。赤を乱用すると警告力が薄まるため、この規律を最後まで守る。

## 実務メモ

公理・定理とは別の、実務的な備忘。

### 技術的前提

- **ベースラインは C++23。** `std::println` など、現代的な書き方を素直にするために C++23 機能を使う。「ベースラインを C++23 にする」＝「C++23 の処理系を前提にしてよい」であって、「C++23 の新機能を網羅する」ではない。
- **バージョン依存の記述は環境構築の章に集約する。** 本文は特定バージョンに縛られない書き方を保ち、更新が必要になったら一箇所だけ直せるようにする。
- 対外的な前提表記は「C++23 以降」とする(旧称「C++20 以降」から更新済み)。

### 本書の content stance

章ごとのスタンスは個々の章本文に書くが、章を横断する設計として次を念頭に置く。

- **関数**は「与えた値から結果を導く処理」として組む見方を主軸に据える。関数の外側に変化を残す働き(副作用)を、関数の主目的にはしない。`std::println` のように作用そのものを役目とする関数の存在は否定しないが、純粋寄りの書き方を中心に置く。
- **変数**は宣言と同時に値と結びつけ、以後はその値を書き換えない使い方を主軸に据える。書き換えが要る場面は別章で扱う。
- これらは「こう書くべき」という押し付けではなく、本書がとる見方として表明する形を保つ(定理の型2に沿う)。
- 副作用を担う関数を使う場面では、その必要性を C++ 言語の制約ではなく、標準ライブラリが代替を提供していないという実用上の話として扱う。「言語上できない」とは書かない。
- **コンストラクタ**は、引数を1つ取るものに原則 `explicit` を付け、意図しない暗黙の型変換を防ぐ(型変換章で導入)。const 正しさと同じ遡及パターンで、導入前の章(constructor-destructor)の `Counter(int)` は非 explicit のまま据え置き型変換章が遡及ノートで回収、導入後の章(polymorphism)の `StepCounter(int)` は explicit に揃えた。

### 編集の作法

- **編集後は通読する**。ファイルを編集した直後に、関連箇所を含めて読み直し、修正が誘発した内部不整合や派生問題がないか確認する。push 前に必ず一度通読の段を挟む。
- **同系統の sweep を欠かさない**。区別や用語を一箇所直したら、同系統がファイル全体に残っていないかを必ず確認する(定理「異なるカテゴリーを区別する」の運用)。
- **章末「宣言の並びとの関係」節は、非宣言の topic を扱う章にのみ置く**。文・型・出力・プリプロセッシングディレクティブ・制御構造・代入・リンクなど、それ自体は宣言でないものを、テーゼ(プログラムを宣言の並びとみる)に繋ぎ直すための節。宣言そのものを扱う章(宣言・関数・変数)には置かない。繋ぎ直すべきカテゴリーの差がなく、置くと「宣言の話に宣言への繋ぎ直しを書く」冗長になるため。この非対称は意図的なものである。

### 章立て(基本編)

基本編は、内部に部の区切りを設けない一続きの入門書とする。前半はアンカーコードを出発点に小さなプログラムを分解し(アンカー駆動)、後半は言語の機能をトピックごとに見ていく(トピック駆動)。手法・規律(2種ボックス・カテゴリーの精密な区別・明示的な先送り・スタンス表明形)は全体を通す。標準ライブラリや基本編で扱いきれない言語機能は、**続きの編**(別途・Zenn で書く前提)で扱うこととし、基本編はおわりにで一区切りとする。**「一通り」「網羅」と読める書き方は避ける**(現状は言語機能の簡単な導入であり、不足を認める)。なお以下の各項に残る「第1部/第2部」は、部を設ける前の名残で、それぞれ前半(アンカー駆動)/後半(トピック駆動)を指す。「第3部」は続きの編を指す。

通しの背骨:宣言の並び(前半)→ その宣言が示す型・言語機能(後半)→ それ自体が型と関数の宣言の集まりであり `#include` で取り込む標準ライブラリ(続き)。

#### 序

- **はじめに**：確定。冒頭に序文(C++ 入門書であること、C++23 を基準とし C++23 以降の環境を前提とすること、新機能網羅は目的としないこと)。本書の主題(ビルドが通ってしまう誤りを作法で防ぐ)、二種類のボックスの約束、対象読者の三層、扱う範囲(古い書き方は意図的に扱わない)。文体は本文と同じ規律(効果予測・読者への要請・心情を乗せない)に揃え済み。
- **環境構築**：後回し。賞味期限が最も短いため、本文が固まってから書く。バージョン依存の記述の集約先。

#### 前半:全体を一巡する(アンカー駆動・確定)

- **宣言**：確定。主題「プログラムを宣言の並びとみる」。アンカーコードを噛み砕く構成。
- **関数**：確定。宣言の章から送った宿題は回収済み。
- **変数**：確定。主題「変数は値に名前を結びつけたもの」。`result` を分解する構成。書き換えは別章送り。
- **式と文**：確定。式と文を整理し、アンカーコード内の各要素を分類する構成。`;` 忘れの黄ボックスを初登場させた。型は別章送り。
- **整数の型**：確定。アンカーコードの `int` の正体を扱う。3 章にわたる伏線(declaration の `int という型の名前`、function の戻り値・仮引数、variable の変数の型、expression-and-statement の式の型)を回収。型はビルド時に検査されるという性質に黄色ボックスを当てた。スコープを `int` に絞るため、章名は「型」ではなく「整数の型」とした。
- **出力**：確定。`std::println` の中身(書式文字列 `{}`、`ln` の意味、`#include <print>`、副作用としての位置づけ)を扱う。`#include` の書き忘れに黄色ボックスを当てた。4 章にわたる伏線(declaration の `std::println による出力`、function の副作用の代表例、variable の値の出力、expression-and-statement の式文の外側への作用)を回収。
- **プリプロセッシングディレクティブ**：確定。`#include` の中身を扱う。`<print>` の取り込み、`< >` vs `" "`(複数ファイル章送り)、プリプロセッシングがビルドの最初の段階であること、その他のディレクティブの名前列挙、C++26 `#embed` の予告を `:::details` コラムで。declaration の「後の章で扱う」と output の「次の章で扱う」を回収。
- **条件分岐**：確定。`if` / `else` による分岐。本書で初めてアンカーコード外のコード(`max` 関数)を導入した章。比較演算子と `bool`(整数の型の章の「論理値を表す型」を回収)、`if` の条件は `bool` 型の式であること(式と文の章の「式は型を持つ」を回収)。`if` の条件の括弧忘れに黄色ボックス。当初「制御構造」として if/while/for を一括予定だったが、繰り返しは書き換えを前提とするため分割した(下記参照)。
- **代入**：確定。variable.md で deferred した「書き換え」を扱う。最大の精度ポイントは初期化と代入の区別(同じ `=` でも、初期化は宣言の一部・代入はそれ自体が操作)。代入は式文であること(式と文章を回収)。本書の不変主軸スタンスを保ちつつ、繰り返しの前提として導入。章名は読者向けの「書き換え」ではなく標準語の「代入」とし、イントロで橋渡し。ボックスは置かない(代入特有のビルド失敗/UB が初学者向けに自然な形で立たないため、box 規律に従い見送り)。
- **繰り返し**：確定。`while` / `for`。ループ変数の書き換えを前提とするため、書き換え/代入の章のあとに置いた。条件は `bool` 式(条件分岐章を回収)、ループ変数の更新は代入(代入章を回収)。`while` で機構を示し、`for` を同じ繰り返しの別形として導入。無限ループは「条件が成り立たなくなる方向へ値が進むことを確かめる」と本文で注意(ビルドは通り、副作用のある無限ループは UB でもないため、色付きボックスは置かず散文で扱う)。`for` の括弧の中の `;` は、本文では「3つの部分」とだけ述べ、厳密(最初の `;` は init-statement の末尾で区切りではない)は `:::details` コラムに逃がした。
- **複数ファイル**：確定。`add` を別ファイルに分け、本体のない宣言と定義の区別を実地で示す(declaration.md L61 の「定義は宣言の一種/複数ファイルで効く」を回収)。ヘッダファイル、`"..."` vs `<...>`(preprocessing-directive.md L19 を回収。規格上はどちらも処理系定義の探索であり「自分のファイル/処理系のファイル」は慣習にすぎない——本文は慣習として述べ、規格の探索規定と `"..."` の `<...>` フォールバックは `:::details` コラムに逃がした)、翻訳単位とリンクを軽く導入。定義のない名前の呼び出し(リンク失敗=ビルド失敗)に黄色ボックス。モジュールは扱わない方針(declaration コラム)に沿い、ヘッダ+ソース分割で構成。
- **翻訳**：草稿。第1部のキャップストーン(複数ファイルの後)。これまで「ビルド」の傘でまとめてきた過程を一度だけ開き、規格の語「翻訳」として preprocess → compile(各翻訳単位)→ link の段階を示す。散らばっていた前処理(preprocessing-directive)・翻訳単位/リンク(multiple-files)を 1 つの像に束ねる。ここで初めて「コンパイル」(段階)を本文で導入(序盤で避けてきた語の回収。以降も本文の常用語は「ビルド」に戻す)。「コンパイラ」(道具)は本文のコンパイル節には置かず、全段階を担うという実態とともにコラムで導入する——道具を 1 段階の節に置くと「コンパイラ=コンパイル段階の道具」という混同を本文側で強めるため。機械語は規格の概念でなく処理系の実態なので本文では断定せず(コンパイルは「解釈され変換される」にとどめる)、実態は `:::details`「翻訳の実態」コラムに逃がした(#include の処理系定義を慣習として扱ったのと同じ流儀)。コラム summary は本文から外した「コンパイラ」を初出にしないため、語を使わず「翻訳の実態」とした。コラムが扱う実態は2点:(1) 実際のコンパイラ(`g++` 等)は前処理・コンパイル・リンクの全段階を 1 つの道具で担うことが多く、「コンパイラ」の名はコンパイル段階に由来するが道具としては翻訳全体を引き受ける(本書の「ビルドツール」≒このコンパイラ)。(2) 規格は翻訳の結果の形を定めないが、多くのコンパイラは機械語にする。本文の段階分け(preprocess/compile/link)は概念整理として保つ。黄ボックス=コンパイル/リンク段階で捕まる誤り、赤ボックス=翻訳をすり抜け実行時に出る誤り、と段階に対応づけて本書の主題に接続。背骨への紐づけ:静的な「宣言の並び」を、それが動くプログラムになる過程の側から回収(構造と過程の両側)。Part 1 確定は崩さず章の追加で対応。

予定の順序は現時点の推奨であり、必要に応じて入れ替える。

**制御構造を分岐と繰り返しに分けた理由**：`while`/`for` は通常ループ変数の書き換えを必要とするが、本書は「変数は宣言時に一度結びつけ、以後書き換えない」を主軸に据え、書き換え/代入を後の章に送っている。分岐(if)は書き換えを前提としないので先に出し、繰り返しは書き換えの導入後に回す。

#### 後半:言語機能(トピック駆動)

**編成原理:型システムを背骨にする。** 第1部の到達点は「宣言は名前を導入し、その型を示す」。第1部が宣言という骨格を描いたのに対し、第2部は、その名前が担う型と、型が決める値・操作を肉付けする。扱うのは言語そのものの機能で、標準ライブラリが提供する型・道具は第3部に送る。各章を束ねる軸は「プログラムを宣言の並びとみる」から「型」へ移る。手法は第1部のまま——具体例から入り、観察して分解し、2種ボックス・明示的な先送り・スタンス表明形を守る。各章は小さな具体プログラムから始め、第1部のアンカー駆動の良さ(具体性)を章単位で取り込む。

以下は候補。第1部の deferred(「必要になる章で扱う」と送った各件)を回収しつつ、型の軸で依存順に並べた。順序・粒度・章名・要否は未確定。

- **基本型**：草稿。第2部の最初の章。`bool`(条件分岐で既出)・`double`(実数)・`char`(文字)を survey し、これらが `int` とともに **基本型**(fundamental type)と呼ばれることを受動形で導入。`double` の `/` が小数の割り算になること(integer-type コラムの回収)。`int` の表せる範囲は処理系依存で、符号付きあふれは UB(赤ボックス)。文字列は標準ライブラリの型なので第3部に送ると明示(言語/ライブラリの分けと整合)。型ごとに値と操作が決まる、で締めて次章(演算)へ。基本型はこれで尽きない旨は exclusion(「本書では深く立ち入らず int/double を主に使う」)で締め、deferral にしない(範囲・精度違いの型は回収先が未定のため。`void`/`std::nullptr_t` は関数・ポインタの章で自然に回収)。冒頭で第1部→詳細編への移行を「ここまで/ここから」で示す(「第2部」の語は本文に出さない)。integer-type が送った「論理値・実数を表す型」を回収。章名の変遷:「型を広げる」(暗黙の前提が多い)→「組み込みの型」(規格用語でない)→ 規格 [basic.fundamental] の語に合わせ「基本型」。
- **列挙型**：草稿(slug: enum)。後半、基本型の次。とりうる値に名前を付けた型(`enum class`)。状態・種類を整数で表すと意味が分からない、を動機に。列挙子(名前付きの値)、`型名::列挙子`、`==` 比較。`enum class` は整数に暗黙変換されない(`int n = Color::red;` →黄ボックス。型安全=取り違え防止の主題に接続)。`class` なしの `enum`(整数へ暗黙変換・名前が漏れる)は使わず enum class を使う(modern)。switch より前に置き、switch 章で enum を分岐に使えるようにする。**「列挙型=とりうる値に名前を付けた型」と定義してから enum class に結びつける**(著者指摘:語を出すだけで定義しない、を避ける)。他の言語との enum 対比(Rust=ヴァリアントがデータを持つ直和型・Java=各値がオブジェクト/C++ は整数に名前を付けたものに限る)を `:::details` コラムに置く。「名前付きの値」では抽象的すぎるので「整数に名前を付けたもの」と具体に。
- **論理演算**：草稿。`&&`/`||`/`!` による `bool` 値の組み合わせ。基本型章の「bool の値どうしを組み合わせる演算は次章で扱う」を回収。条件分岐(比較のみ)を複数条件の組み合わせへ拡張。`1 <= x <= 10` が意図どおりにならない件を `&&` の動機に。短絡評価(左で決まれば右を評価しない)を簡潔に。当初「演算と論理」で複合代入・シフトも一括予定だったが、論理とは別系統(算術の略記・ビット操作)で焦点がぼやけるため論理演算に絞った。複合代入(`+=`/`++`)・シフトは別途(下記)。
- **その他の演算と複合代入**：草稿(slug: more-operators)。第2部、論理演算の次。演算スレッド(整数の型=算術 → 条件分岐=比較 → 論理演算=論理 → 本章)の締め。**順序は「その他の演算 → 複合代入」**(複合代入 `op=` は二項演算子と `=` の合成なので、先に演算子を出し切ると `&=` 等も一般形で収まる、という依存)。**ビット演算・シフト**(`&`・`|`・`^`・`~`・`<<`・`>>`)= integer-type L21「他の演算は必要になる章で扱う」の回収先。intro なので軽く survey し「ビット操作は主筋にしない」と exclusion で締める(深掘りの回収先はない)。**複合代入**:`x op= y` は `x = x op y` の短縮、`++`/`--` も。loop の `i = i + 1` を `i += 1`/`i++` で回収、代入章の「書き換え」の一種と位置づける。前置 `++i`・後置 `i++` を両方紹介。式の値の違い(後置=増やす前の値/前置=増やした後の値)を簡潔に例示し、値を使わない文の範囲ではどちらも同じ、と述べる(複雑ではないので exclusion にせず説明する)。ボックスは置かない(自然なビルド失敗/UB が立たないため)。まとめ節も置かない。
- **const**：草稿。型に「書き換えない」という性質を付す。variable の方針「書き換えない」とコラム「既定で書き換え可/禁じる手段は別」を回収。const な変数への代入はビルド失敗(黄ボックス)、const は初期化必須(初期化子なしもビルド失敗)。「方針を言語に守らせる(方針→保証)」で締め。型の背骨に紐づけ(const は型に付く性質)。表記は east const(`int const`)に統一(定理化済み)。本章で「前置 `const int` も同じ意味だが本書は後置に統一する」と一度断る。`int const` の部分が `decl-specifier-seq` であり `const` はその一要素で、並びの順序は問われないため east/west が等価——という文法的裏づけは `:::details` コラムに逃がした(第1部の宣言の枠組みと地続き)。章名は仮に「const」(キーワードそのもの)。「定数」は規格上 const 変数と一致しないため避けた。要検討。
- **クラス**：草稿。第2部の山場の前半。自分で定義する型は **クラス**(規格語。`struct`/`class` で定義)であり、本章では `struct` キーワードを使う。複数の値(メンバ)をまとめ、メンバ関数(型に属する関数)まで扱う。declaration 章の「多くの宣言は名前(値・関数)を導入する」に対し、クラスの定義は型を導入する宣言である、と示す(declaration.md にこの具体句は無いので、偽の引用参照をしない。declaration が述べた『名前を導入する』に紐づける)。基本型章「型は値と操作を定める」を、自分で値(メンバ)と操作(メンバ関数)を決める形で回収。メンバ初期化は `= {...}`(コピーリスト初期化)で変数章の `=` と一貫させる(直接リスト初期化 `r{...}` は使わない)。**指示付き初期化**(`= {.width = 3, .height = 4}`)を本筋に据え、位置指定(`= {3, 4}`)は「素の形だが宣言順に依存し、`{4, 3}` と取り違えてもビルドは通り誤った値が入る」として併記。名前で指定すれば取り違えを防げる=本書の主題(作法で誤りを防ぐ)に合う。`= {...}`(波括弧での初期化=リスト初期化)が使えるか・何を意味するかは型による。集成体では集成体初期化(メンバ単位)になり、`initializer_list` 等を取るコンストラクタを持つ型ではそのコンストラクタ呼び出しになるなど、`= {...}` は集成体に限らない(「集成体限定」「コンストラクタを持つと `= {...}` でなく別の書き方」は誤り。指示付き初期化 `{.x = ...}` のほうは集成体限定だが、その厳密も本文では出さない)。規則が複雑なので本文は言い切らず、「`= {...}` が使えるか・何を意味するかは型の定め方による/Rectangle ではデータメンバを初期化する」と hedge する。経験者向けに集成体の詳細を `:::details`「集成体とリスト初期化」コラムに補う:集成体になる条件は例示にとどめ(コンストラクタ不在・データメンバ公開など)、正確な条件は規格・版を追って変わってきた、とだけ述べる(変遷の具体は挙げない)。仮想でないメンバ関数は妨げないが、仮想関数を持つ型は集成体にならない(仮想関数は集成体の条件「仮想関数の不在」に触れる)。集成体・コンストラクタの語は本文に出さない。`.` でのアクセス、定義末尾の `;` 忘れに黄色ボックス。メンバは値ではなく値を保持する名前付き要素として扱い、メンバ関数もメンバの一つと位置づける(メンバ=値、メンバ≠関数 の混同を避ける)。総称は「メンバ」、その下位が **データメンバ**(規格語 data member。基本とする)と **メンバ関数**。データメンバの別名「メンバ変数」も一度だけ紹介(受動形)。width/height はデータメンバ、area はメンバ関数。アクセス制御は扱わず次章へ commit。章名は当初「構造体」としたが、規格では struct で定義した型も class(クラス)であり「構造体」という型カテゴリは規格に無いため、概念を規格語「クラス」で立て struct はそれを定義するキーワードと位置づけた(基本型=fundamental type と同じく [[feedback_prefer_standard_term]] の運用)。メンバ関数の節末に他言語比較コラム(データと操作の結びつけ方:C の struct はデータのみで関数なし/Rust は操作を別の impl ブロックに置く/C++ はクラス定義に両方。加えて C との部分的互換=データだけの構造体は同配置で相互運用できるが C++ 独自機能で崩れる。仮想関数など未習語は出さず「C++ 独自の機能」と一般化し「ことがある」で hedge)。
- **デフォルト初期化**：草稿。第2部、クラスとアクセス制御の間に挿入(slug は default-initialization)。初期値を書かずに変数・オブジェクトを作る=デフォルト初期化。組み込み型(`int result;`)は値が定まらない(変数章の未初期化を回収・深掘り)、クラスもデータメンバにデフォルトメンバ初期化子がなければ同じ(`Rectangle r;` で width/height が定まらない)→ 値の定まらないメンバ読み取りに赤ボックス。**デフォルトメンバ初期化子**(`int width = 0;`)で安全になる、を導入。独自の初期化=コンストラクタは後続、とだけ触れる。集成体の話はクラス章のコラムに残し、本章はデフォルト初期化に絞る(ユーザー選択)。これによりアクセス制御章は概念を使うだけで済む。

  **「default」の訳語(ユーザー選択A)**：名前付きの C++ 概念(default-initialization / default member initializer)は確立した「デフォルト〜」(デフォルト初期化・デフォルトメンバ初期化子。cppreference・本書のカタカナ系統に合致)。一般の「by default」(変数章「既定で書き換えられる」、struct/class の「既定のアクセス」)は「既定」のまま。register で使い分ける。
- **アクセス制御**：草稿。山場の後半。`public`/`private`、`struct`/`class` の違い(既定のアクセス。メンバの既定アクセスは本章、継承における既定アクセスの違いは継承章送り。「メンバの既定アクセスだけ」は継承にも現れるので不正確)、private アクセスのビルドエラー(黄ボックス)、カプセル化。クラス章から deferred したアクセス制御・struct/class の違いを回収。コンストラクタは使わず(ユーザー選択でアクセス制御に絞る)、既定値を持つ private 状態 + public メンバ関数の例 `Counter`(新規。Rectangle は private 化すると集成体でなくなり初期値にコンストラクタが要るため使わない)。デフォルトメンバ初期化子・デフォルト初期化は前章で導入済みなので本章は使うだけ(`count = 0` はデフォルトメンバ初期化子、`Counter c;` はデフォルト初期化、と前章を参照)。カプセル化は「データを隠すかどうかは設計の判断」と述べる(本書は public struct も使うので「カプセル化を主に用いる」とは断定しない=over-claim 回避)。private は「クラスの内側からしか使えない」と簡略化(friend は範囲外)。アクセス指定子は public/private/protected の3つなので、protected を無いことにせず一文で acknowledge する。「本書で扱わない」と締め切るのは尚早なので、継承に関わるものとして「後の章で触れる」と deferral にし、回収先として継承を roadmap に立てた(下記)。コンストラクタは後続章へ。
- **型と関数**：草稿。第2部、アクセス制御の次(slug: type-and-function)。第1部の関数章を型の観点から拡張。中心の見方「関数は引数の型と戻り値の型で特徴づけられる」。内容:**関数の型**(`int(int, int)`。戻り値の型+仮引数の型、名前なし)と**シグネチャ**の区別(シグネチャ=名前+仮引数の型、戻り値の型を含まない=通常の関数の場合。「からなる」でなく「含む」で非 exhaustive に書く。対象依存=テンプレートは戻り値の型・メンバ関数はクラス/const 修飾、はコラムで吸収。名前空間は既出の `std::` に錨を打って導入回避)、**値渡し**(コピー、呼び出し側を変えない→値を返す形と整合)、**オーバーロード**(シグネチャで区別/戻り値の型だけの違いは不可。シグネチャが戻り値の型を含まない→区別されない、の向きで因果を書く。名前マングリングは処理系/ABI の実態としてコラムに隔離=機械語コラムと同流儀)、**デフォルト引数**。**参照渡しは含めない**(参照未導入のため参照とポインタ章へ)。**ラムダ・再帰も含めない**(どちらも続きの編へ。ラムダは常用先のアルゴリズム等と同じ編、再帰は基礎には負荷が高いため)。シグネチャコラムで「メンバ関数では const 修飾が加わる」と次項の布石を置いてある。
- **参照とポインタ**：草稿(slug: reference-and-pointer)。第2部、型と関数の次。他のオブジェクトを指す2つの仕組み。**参照**=既存オブジェクトの別名(新しいオブジェクトを作らない・それ自体はオブジェクトではない)。対象への結びつけを **束縛** と呼び、いったん束縛した参照は **再束縛できない**(`r = y;` は参照先に書き込むだけ)。初期化子なしの参照→ビルド失敗(黄ボックス)。**参照渡し**(`int&`)で呼び出し側を書き換えられる、**const 参照**(`Rectangle const&`)でコピーを避けて読み取る(型と関数章「大きなオブジェクトのコピーを避けたい」の回収)。const 参照越しの書き換え→ビルド失敗(黄ボックス)。**ポインタ**=アドレスを値として持つオブジェクト(それ自体がオブジェクト型=参照との違い)、`&`/`*`、**指す先を変えられる**(参照=束縛/ポインタ=指す先を変える、で語を割る → feedback_binding_term_reference_only)、`nullptr`。null 越しのアクセス→実行時 UB(赤ボックス)。ポインタの深い扱いは exclusion(本書では立ち入らない)。オブジェクト型の否定形定義は本文で言い切らず必要な事実だけ引く(参照型はオブジェクト型でない/ポインタはオブジェクト型である → feedback_verify_standard_claims)。binding 語(束縛/再束縛)は参照に限り、ポインタには当てない。
- **オブジェクト引数と const メンバ関数**：草稿(slug: object-parameter)。第2部、参照とポインタの次。const メンバ関数の前に、メンバ関数が `.` の左のオブジェクトを隠れた引数として受け取る**オブジェクト引数**を導入する(const 修飾はオブジェクト引数に効くので、これを先に置かないと const の意味が立たない)。`area` が引数を書かずに `width`/`height` を使えた事実を起点にし、型と関数章のシグネチャコラムの布石(メンバ関数では const 修飾が加わる)を「その const はオブジェクト引数に効く」と回収。規格コラム:**暗黙のオブジェクト引数**(implicit object parameter)はオーバーロード解決上の概念、C++23 の明示的オブジェクト引数(`this` 付き仮引数)に言及し本書は暗黙のみ扱う(呼称・C++23 機能とも著者が裏取り済み)。**const メンバ関数**=仮引数並びの後ろに `const`、オブジェクト引数を通して書き換えない約束。参照とポインタ章の **const 参照** に接続(const 参照で書き換え不可と同型)。黄ボックス:const メンバ関数内でデータメンバ書き換え→ビルド失敗。const オブジェクト/const 参照の先では const メンバ関数しか呼べないことを、前章の const 参照を使う `print_area` で動機づけ(read-only な `area` を const にすれば const 参照の先でも呼べる)。最後に read-only な `area`(クラス章)・`value`(アクセス制御章)を一斉に const 化(本来 const にすべきものを当面非 const に揃えてあった deferral の回収。片方だけの piecemeal はしない)。**const correctness は「const 正しさ」と訳さない**(造語で機械翻訳臭、と著者指摘 → feedback_no_coined_translation)。「英語では const correctness と呼ばれる」と英語呼称を一度示し、あとは本書の作法(書き換えない操作に const)として述べる。オブジェクト引数の導入例で `c.value()` のローカル未定義 `c` を先取りしない(`r.area()` の一例に閉じる)。
- **コンストラクタとデストラクタ**：草稿(slug: constructor-destructor)。第2部、オブジェクト引数の次・継承の前。クラス章・アクセス制御章・デフォルト初期化章で「独自の初期化=コンストラクタは後続」と送った件の回収先。中心は **コンストラクタ**(オブジェクト生成時の独自初期化。型と関数章のオーバーロード・デフォルト引数が使える位置)。例は `Counter` を 0 以外から始めたい動機で `Counter(int start) : count(start)` を導入。**メンバ初期化子リスト** は `count(start)` の部分で、`:` は含まない(導入記号として別に説明。著者指摘:mem-initializer-list はコロンを含まない)。**デフォルトコンストラクタ** `Counter()`:`Counter(int start)` を書くと `Counter c;` がビルド失敗(黄ボックス)→ 引数なしのコンストラクタを足す。ここは**具体例に閉じる**——「コンストラクタが1つもなければ暗黙に用意される」の一般則は delete 例外があり断定すると型1に触れるため、観察事実(以前書けた `Counter c;` が `Counter(int start)` を書いたら使えなくなった)だけ述べる(→ feedback_verify_standard_claims)。**メンバ初期化子リストとデフォルトメンバ初期化子の関係**:list に書いたメンバはその値、書かなかったメンバはデフォルトメンバ初期化子の値、どちらもなければデフォルト初期化(int は値定まらず=デフォルト初期化章の回収)。コラムで初期化子リスト vs 本体代入(本体で `count = start` は「デフォルト初期化されてから代入」=初期化と代入の違い、代入章)。**デストラクタ** `~Counter()` は寿命の終わりに呼ばれる対の概念として導入(`{ }` ブロックを抜けるとき走ることを `std::println` で観察、出力は確認のためと明示)。後始末が「本当に要る」は主観なので「関わってくる」に(型2 → feedback_no_subjective_adjectives)。**new/delete・仮想デストラクタ・資源の後始末/RAII は扱わず** new/delete 章(多態の後ろ)へ送る。重心はコンストラクタ寄り、デストラクタは概念導入にとどめる。この章を継承・多態の前に置くことで、多態章の「仮想デストラクタを付けておくとよい」が、デストラクタ既知の上での言及になる(cold な語の先取りを避ける)。
- **型変換**：草稿(slug: conversion)。第2部、コンストラクタとデストラクタの次・継承の前。明示的な変換から入り暗黙の変換へ(著者方針:明示先行)。**`static_cast<型>(式)`** で明示変換(`double`→`int` は小数部切り捨て)。**暗黙の変換**(`int n = 3.9;` が static_cast なしで黙って `3`)を「ビルドの通る誤り」として主題に接続(散文、box なし=ビルド通る・UB でない)。**縮小変換** + 波括弧:`int n = {3.9};` はビルド失敗(黄ボックス)、意図するなら `static_cast` で明示。**変換コンストラクタと `explicit`**:`Counter c = 5;` が `Counter(int)` 経由で暗黙変換(=変換コンストラクタ)、`explicit` で禁じる(黄ボックス)。派生→基底の変換は本章では扱わない(値の変換に絞る。多態章の「指せる」のまま)。
- **継承**：未着手(設計確定、slug: inheritance)。第2部、型変換の次。クラスの章群(クラス・アクセス制御)の続き。主目的はアクセス制御章からの deferral 回収:**protected**(章 L9)と、`struct`/`class` の継承における既定アクセスの違い(struct は public 継承・class は private 継承が既定。章 L61)。最小限の **派生クラス** を導入する。例は `Counter` を土台にし、`count` を private → **protected** に上げ、派生 `ResettableCounter : public Counter` が `reset()` で `count = 0` にする(private では派生から触れず protected なら触れる、で protected の存在理由が立つ。Rectangle/Counter 使い回しの継続性)。基底の private メンバへ派生から触れる→ビルド失敗(黄ボックス)、protected メンバへ外から触れる→ビルド失敗(黄ボックス)。継承の既定アクセスは `: public`/`: private` を明示する作法を勧め、省略時の既定が struct=public・class=private と異なることを述べる(private 継承の深い意味には立ち入らず hedge)。**スタンス**:本書は単純な型・合成寄りなので、継承は「そういう仕組みがある」と回収範囲にとどめ過度に推奨しない。**多態(polymorphism)は触れるが本章では扱わない**——重い別主題(`virtual`・オーバーライド・スライシング・仮想デストラクタ)なので独立章へ deferral(下記を回収先に立てる。宙吊りの exclusion にしない)。
- **多態(仮想関数)**：草稿(slug: polymorphism)。継承章から deferral した polymorphism の回収先。継承・参照とポインタ・コンストラクタとデストラクタが前提。核:基底クラスに **仮想関数**(`virtual`)、派生で `override` を付けて **オーバーライド**、基底クラスの **参照・ポインタ** を通して呼ぶと **動的な型** に応じて呼び分けられる(動的ディスパッチ)。例は Counter 拡張:基底 `Counter`(継承章で count を protected 化済み)の `increment` を `virtual`、派生 `StepCounter(int)` が step ぶん増やすよう override、`count_up(Counter&)` に Counter/StepCounter を渡して `2`/`20` で呼び分け。「基底クラスの参照は派生クラスのオブジェクトを指せる」は本章初出の事実なので「参照渡しなので」と既知から滑り込ませず、public 継承を根拠に立てる(→ feedback_introduce_new_facts)。**`override` の安全性を本書の主題に直結**させて入れた:`incrment()` の綴り間違いに `override` を付けるとビルド失敗(黄ボックス)、付けないと別関数になり基底版が呼ばれ続ける silent bug=「ビルドの通る誤りをビルド段階の誤りに変える」。**スライシング**(`Counter c = b;` で派生部分が落ちる)は散文で示す(ビルドは通り UB でもないので黄でも赤でもない=box 規律)。**仮想デストラクタ**:多態の基底には virtual にしておくとよい、とスタンス表明で触れ(「望ましい」を具体理由=基底ポインタ経由の破棄に紐づけ型2回避)、なぜ効くか=new/delete は new/delete 章へ先送り(デストラクタは ctor 章で既知なので cold でない)。**部分型付け(subtyping)は本文に出さない**(型理論語で規格語でなく、置換可能性を保証しないため over-claim=型1。経験者向け最小コラムの余地はあるが現状省く)。ポインタ経由のデモは new/delete を呼び込むため避け、本文は参照中心+ポインタは言及のみ。
- **動的なオブジェクト(new/delete)**：草稿(slug: new-delete)。多態章の次・配列章の前。コンストラクタとデストラクタ章で送った「資源の後始末・RAII」と、多態章で送った「仮想デストラクタがなぜ要るか」の回収先。`new`/`delete` による動的なオブジェクトの確保と破棄、基底クラスのポインタを通して派生オブジェクトを破棄する場面、そこで仮想デストラクタがないと深刻な問題が起こりうること(赤ボックス候補)。delete 忘れによる leak など手動管理の落とし穴を見せたうえで、**第3部への橋渡し**として、確保と破棄を引き受ける標準ライブラリの道具 `std::unique_ptr`・`std::shared_ptr` の名前を出す(`std::vector` をクラス章コラムで先取りしたのと同じ「語を出すが定義は送る」。回収先は第3部スマートポインタ項)。粒度・章名・置き場は未確定だが、上記の先送りの回収先として立てる(dangling にしない)。
- **配列**：草稿(slug: array)。第2部、new/delete の次。同じ型の値を並べて扱う言語機能。固定長 `int a[5]`(`= {...}` 初期化・添字 `a[i]`・0 始まり・`for` の添字ループで反復)、**範囲外アクセスは UB**(赤ボックス。配列での主題の核)、動的な配列 `new int[n]`/`delete[]`(new/delete の延長。配列名・`new[]` の戻り値が先頭を指す=ポインタとの関係を軽く)。new/delete →スマートポインタと対称に、生の配列=基礎の機構/`std::array`(固定)・`std::vector`(可変)=道具(第3部)へ橋渡し。範囲 for は第3部に据え置き、本章の反復は添字ループ。まとめ節は置かない。modern スタンス:生配列は推奨せず道具へ送る立て付け。
- **break と continue**：草稿(slug: break-continue)。第2部、その他の演算と複合代入の次(命令的な基礎=演算子・制御フローを型システムの前にまとめる位置。loop/branch=第1部が前提で型システム依存はない)。`break`=繰り返しを途中で抜ける(最初に7で割り切れる数を見つけたら止める例)、`continue`=今回の本体を飛ばして次へ(奇数だけ出力する例)。例は `for`(`i++` は more-operators で既出)。`continue` の精密:`for` の更新は continue 後も行われる/`while` は条件へ直接戻るので本体内の更新を飛ばすと終わらない繰り返しになりうる(loop 章の無限ループ警告に接続、散文)。範囲 for はコンテナと結びつくため第3部に据え置き。ボックス・まとめ節なし。
- **switch**：草稿(slug: switch)。後半、break-continue の次(制御構造の締め)。一つの値を case ごとに振り分ける。整数の例(`case`/`default`/`break`)+ 列挙型でも分けられる(`case Color::red:`、enum 章へ tie)。`break` は break-continue 章のものと同じ語で switch を抜ける、と接続。**`break` 忘れ=次の case へ流れ落ちる**(ビルドは通るが意図と違う=主題に接続、散文。フォールスルーは UB でないので box なし)。まとめ節なし。
- **auto**:草稿(slug: auto)。終盤、array の次・outro の手前。明示の型を一通り教えた後に「初期化子から型を推論させる」を導入(型のスパインを先に立てる狙い)。`auto n = 5`(int)・`auto d = 3.14`(double)=型推論。**静的に一つに決まる**(動的型付けではない/どんな型でも入る、ではない)を本文で打ち消す。初期化子必須=黄ボックス(`auto n;`)。長い型名で省ける例 `auto p = new Counter();`(動的なオブジェクト章の `Counter* p = new Counter();` と対比)。標準ライブラリの長い型名で特に効く点は続きの編。締め=明示と auto は同じ型、型を見せたいとき明示・初期化子から明らかなとき auto。まとめ節なし。

#### 続きの編(別途/Zenn):ライブラリ機能ほか

**編成原理:標準ライブラリは、あらかじめ用意された宣言の集まり。** 標準ライブラリは、型(`std::vector`・`std::string` など)と関数(`std::println` など)の宣言をまとめたもので、`#include` で取り込んで使う。これは第1部の宣言の見方・出力章の `std::println`・複数ファイル章のヘッダ取り込みと地続きである。第1部・第2部で作った型と宣言の見方の土台の上に、標準ライブラリの道具を載せる。手法(具体例から入る・2種ボックス・明示的な先送り・スタンス表明形)は据え置き。

続きの編では、基本編から送った **言語機能** と、**標準ライブラリ** を扱う。以下は候補で、順序・粒度・章名・要否は未確定。

言語機能(基本編から送ったもの):
- **再帰**：関数が自分自身を呼ぶ。停止条件の設計・呼び出しの積み重なり・止まらない再帰のスタック消費(ビルドは通るが実行時に問題)など、基礎の導入には負荷が高いため基本編から送った。
- **ラムダ**：無名の関数・キャプチャ(値/参照)。常用先がアルゴリズム等なので、それらと同じ編で導入するのが自然(型と関数章でも別章送りにしていた)。
- **例外**：`try`/`catch`/`throw` によるエラー処理。

標準ライブラリ:
- **文字列**：`std::string`。integer-type が送った「文字列を表す型」の最終回収先。
- **コンテナ**：`std::vector`(可変長)・`std::array`(固定長)など、値を並べて持つ型。配列章からの橋渡しの回収先。範囲 for もここで扱う。
- **出力の詳細**：`std::println` の書式指定(桁数・進数など)。output.md が「後の章で扱う」と deferral した件の回収先(exclusion から deferral に改めた)。
- **入力**：標準入力からの読み取り。
- **スマートポインタ**：`std::unique_ptr`・`std::shared_ptr`。動的なオブジェクトの確保と破棄を RAII で引き受ける道具。new/delete 章が「続きの編で扱う」と送った名前の回収先。手動の new/delete に対する現代的な代替として位置づける。
- (アルゴリズムなど、さらなるライブラリ機能は必要に応じて。)

### 各章の申し送り(回収済み)

### 保留事項

- 本書の中心テーゼ「プログラムを宣言の並びとみる」の射程は、モジュールを使わない翻訳単位に限られる。C++20 モジュールを使う翻訳単位は `module-declaration` やモジュールフラグメントを含み、`declaration-seq` だけでは説明しきれない。この点は declaration.md のコラムに明記済み。本文のテーゼは断定形(「〜である」)を避け「〜とみる」というスタンス表明形に統一した(定理の型2)。将来モジュール章を起こすときに、この射程を改めて扱うか検討する。
