---
title: "オーバーロード解決"
---

基本編の型と関数の章で、同じ名前のまま引数の違う関数を複数定義できることを見た。これをオーバーロードと呼んだ。その名前を呼んだとき、どの関数が呼ばれるかを決める手続きを **オーバーロード解決** と呼ぶ。本章では、複数の候補から呼ばれる関数が選ばれる過程と、選べないときにビルドに失敗することを扱う。

## 候補から選ぶ

引数の型でオーバーロードした `f` を考える。

```cpp
void f(int x)    { std::println("int: {}", x); }
void f(double x) { std::println("double: {}", x); }
```

`f` を呼ぶと、次のように呼び分けられる。

```cpp
f(5);     // int: 5
f(2.5);   // double: 2.5
```

`f(5)` では `f(int)` が、`f(2.5)` では `f(double)` が呼ばれる。この選び方は、おおよそ次の順に決まる。

まず、その名前で見つかる関数が **候補** になる。ここでは `f(int)` と `f(double)` の2つである。次に、候補のうち、与えた実引数を受け取れるものが残る。引数の数が合い、各実引数がパラメータの型へ変換できる候補である。最後に、残った候補の中から、実引数の当てはまり方が最もよいものが選ばれる。

`f(5)` では、実引数 `5` は `int` である。`f(int)` はこれをそのまま受け取れる。`f(double)` も `int` から `double` への変換を介せば受け取れる。変換の要らない `f(int)` のほうが、変換を要する `f(double)` より優先され、`f(int)` が選ばれる。`f(2.5)` では、逆に `f(double)` がそのまま受け取れるので、`f(double)` が選ばれる。

## 受け取れる候補がないとき

どの候補も実引数を受け取れないと、呼べる関数がない。次の `Point` を、`int` と `double` でオーバーロードした `k` に渡してみる。

```cpp
struct Point
{
    int x;
    int y;
};
```

```cpp
void k(int x);
void k(double x);
```

`Point` は `int` にも `double` にも変換できない。

:::message
受け取れる候補のない呼び出し
```cpp
Point p = {1, 2};
k(p);
```
`Point` は `int` にも `double` にも変換できないので、`k(int)` も `k(double)` も `p` を受け取れない。呼べる候補がなく、ビルドに失敗する。
:::

## 候補を一つに絞れないとき

逆に、複数の候補が同じ程度に当てはまることもある。

```cpp
void g(long x);
void g(double x);
```

`g(5)` を考える。実引数 `5` は `int` である。`int` から `long` への変換でも、`int` から `double` への変換でも `5` を受け取れる。どちらも同じ程度の変換で、一方がもう一方より当てはまるとはいえない。候補を一つに絞れないこの呼び出しを **曖昧** と呼ぶ。

:::message
候補を一つに絞れない呼び出し
```cpp
g(5);
```
`int` から `long` への変換と `int` から `double` への変換は同じ程度で、`g(long)` と `g(double)` のどちらがより当てはまるとはいえない。候補を絞れず、ビルドに失敗する。
:::

曖昧な呼び出しは、`static_cast<long>(5)` や `static_cast<double>(5)` のように変換を明示すると、どちらの候補に当てるかが定まる。

:::details 変換の良し悪しの段階
候補の「当てはまりの良さ」は、実引数をパラメータの型へ受け取るのに要する変換の良し悪しで決まる。変換には段階があり、おおまかには、変換が要らない(完全一致)のが最もよく、次に `char` から `int` のような昇格、その次に `int` から `long` や `int` から `double` のような変換、と続く。より良い段階で受け取れる候補が優先される。

`f(5)` で `f(int)` が選ばれたのは、`int` をそのまま受け取る完全一致が、`int` から `double` への変換より良いからである。一方 `g(5)` では、`int` から `long` も `int` から `double` も同じ「変換」の段階にあり、優劣がつかない。これが曖昧になる理由である。
:::

## テンプレートも候補になる

候補には、関数だけでなく関数テンプレートも入る。同じ名前で、関数テンプレートと、テンプレートでない関数の両方があるとする。

```cpp
template <typename T>
void h(T x) { std::println("template: {}", x); }

void h(int x) { std::println("non-template: {}", x); }
```

```cpp
h(5);     // non-template: 5
h(2.5);   // template: 2.5
```

`h(5)` では、`h(int)` がそのまま当てはまり、`T` を `int` とした `h` も同じくそのまま当てはまる。どちらも同じ程度に当てはまるとき、テンプレートでない関数のほうが優先され、`h(int)` が選ばれる。これは、テンプレートとテンプレートでない関数が同じ程度に当てはまるときの決まりである。`g(5)` のようにテンプレートでない関数どうしが同じ程度のときは、この決まりは働かず、曖昧になる。`h(2.5)` では、`h(int)` は `double` から `int` への変換が要るのに対し、`T` を `double` とした `h` はそのまま当てはまる。今度はテンプレートのほうが当てはまるので、そちらが選ばれる。

関数テンプレートは、実引数から型を決められれば候補になる。このとき、その型で本体が書けるかどうかは問われない。本体が書けない型でも候補として選ばれ、選ばれた後の実体化でビルドに失敗することがある。テンプレートの章の `twice` に `+` を持たない型を渡した場合が、これである。
