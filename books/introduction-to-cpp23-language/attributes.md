---
title: "属性"
---

`[[...]]` の形で宣言などに付ける付加情報を **属性** と呼ぶ。属性の多くは、実装が無視してもよいヒントである。本章では、誤りをビルドツールに指摘させる診断系の属性、`[[nodiscard]]`・`[[maybe_unused]]`・`[[fallthrough]]` を扱う。

## [[nodiscard]]

値を返す関数の戻り値を、使わずに捨ててしまうことがある。副作用のない関数では、戻り値を捨てる呼び出しの多くが書き間違いである。関数に `[[nodiscard]]` を付けると、その戻り値を捨てたとき、ビルドツールが指摘する。

```cpp
[[nodiscard]] int twice(int x)
{
    return x + x;
}
```

```cpp
twice(5);
```

`twice(5);` は、戻り値を使わずに捨てている。`twice` は `[[nodiscard]]` なので、ビルドツールはこれを警告として指摘する。ビルド自体は成功するが、捨ててよいのかを確かめる手がかりになる。戻り値を使えば、警告は出ない。

```cpp
std::println("{}", twice(5));
```

ビルドが通ってしまう誤りを作法で防ぐ、という主題からすれば、戻り値の捨て忘れもその一つである。`[[nodiscard]]` は、それをビルドツールの指摘へと変える。

## [[maybe_unused]]

逆に、意図して使わないものもある。使わない変数や引数があると、ビルドツールは未使用だと警告することがある。意図して使わないと示すには、`[[maybe_unused]]` を付ける。

```cpp
[[maybe_unused]] int debug_value = twice(5);
```

`debug_value` を使わなくても、`[[maybe_unused]]` が付いていれば、未使用の警告は出ない。意図して使わないことを、書き手が示している。

## [[fallthrough]]

基本編の switch の章で、`case` に `break` を書かないと次の `case` へ流れ落ちる、と見た。多くは書き忘れだが、意図して流すこともある。ビルドツールは、この流れ落ちを警告として指摘できる。意図して流すときは、`[[fallthrough]];` と書いて意図的だと示す。`[[fallthrough]];` の直後は、次の `case`(または `default`)でなければならない。

```cpp
switch (n)
{
case 1:
    std::println("1");
    [[fallthrough]];
case 2:
    std::println("2");
    break;
}
```

`[[fallthrough]];` を置くと、直後の `case 2` へ流れ落ちるのが意図的だと示され、警告は出ない。これを書かずに流れ落ちる `case` は、ビルドツールが指摘できる。`break` の書き忘れによる、意図しない流れ落ちを見つける手がかりになる。

:::details 属性の無視性(C++26)
属性は実装が無視してよいヒントだと言われるが、「無視」が何を指すかは C++20 まで曖昧だった。C++26 で、これが3つの規則に整理された(P2552R3)。

- 構文は無視できない。標準属性の書き方が誤っていれば(たとえば、引数を取らない属性に引数を渡すなど)、実装はそれを診断しなければならない。
- 意味は無視できることがある。適格なプログラムから、ある標準属性をすべて取り除いても、残る振る舞いが元の適合動作の一つであれば、観測できる振る舞いを変えなくてよい。本章の `[[nodiscard]]` や `[[maybe_unused]]` も、取り除けば警告が出なくなるだけで、この意味で無視できる。メモリレイアウトの章の `[[no_unique_address]]` も、取り除くとオブジェクトが大きくなりうるが、それも適合動作である。
- 機能テストの `__has_cpp_attribute` は、その属性の意味まで実装しているときだけ正の値を返す。
:::
