---
title: "コンセプト"
---

テンプレートは、型をパラメータにして、多くの型に使える宣言を一つ書く。だが、その本体が扱えない型を与えても、書いた時点では何も示されない。実際にその型で使ったとき、テンプレートの本体の中でビルドに失敗する。**コンセプト** は、テンプレートのパラメータが満たすべき要求を表し、使う時点でそれを確かめるしくみである。本章では、コンセプトで要求を表し、合わない型を使う側で退けることを扱う。

## 合わない型でテンプレートを使う

テンプレートの章の `twice` を振り返る。

```cpp
template <typename T>
T twice(T x)
{
    return x + x;
}
```

`twice` は `x + x` を返す。`int` や `double` には使えるが、`+` を持たない型には使えない。次の `Point` には `+` がない。

```cpp
struct Point
{
    int x;
    int y;
};
```

この `Point` を `twice` に渡してみる。

```cpp
Point p = {.x = 1, .y = 2};
twice(p);
```

`twice(p)` と呼ぶと、`T` を `Point` としたテンプレートから関数が作られる。これは、オーバーロード解決の章で見た実体化である。このとき本体の `x + x` は `Point` どうしの `+` になるが、`Point` に `+` はないので、ビルドに失敗する。

:::message
`+` を持たない型でテンプレートを実体化する
```cpp
Point p = {.x = 1, .y = 2};
twice(p);
```
`twice(p)` は `T` を `Point` として実体化され、本体の `x + x` が `Point` では書けない。ビルドに失敗する。
:::

このとき失敗する場所は、呼び出した `twice(p)` ではなく、本体の `return x + x;` である。`twice` が `T` に何を求めるかは、本体を読むまで分からない。

## コンセプトで要求を表す

`twice` が `T` に求めるのは、「`x + x` が書けること」である。これをコンセプトとして書く。

```cpp
template <typename T>
concept Addable = requires (T x)
{
    x + x;
};
```

`concept` は、型が満たすべき要求に名前を付ける宣言である。ここでは `Addable` がその名前になる。`=` の右は、`T` が要求を満たすかどうかを表す、ビルドの時点で定まる条件である。

`requires (T x) { x + x; }` を **requires 式** と呼ぶ。`{ }` の中に、その型でできてほしい式を並べる。ここに書いた `x + x` が `T` で組み立てられるなら、この requires 式は満たされる。確かめるのは式が書けるかどうかであって、`x + x` の値が何になるかは問わない。`Addable` は、「`x + x` が書ける型」を表すコンセプトになる。

## テンプレートを制約する

コンセプトを使って、`twice` のパラメータに制約を付ける。`typename` の代わりにコンセプトの名前を書く。

```cpp
template <Addable T>
T twice(T x)
{
    return x + x;
}
```

`template <Addable T>` は、「`Addable` を満たす型 `T`」という意味になる。`int` や `double` は `Addable` を満たすので、これまでどおり使える。

```cpp
twice(5);     // int は Addable を満たす
twice(2.5);   // double も満たす
```

`Point` は `Addable` を満たさない。制約を満たさない型では、その候補が受け取れる候補から外れる。`twice` はほかに候補がないので、`twice(p)` は呼べる関数がなくなる。

:::message
制約を満たさない型でテンプレートを呼ぶ
```cpp
Point p = {.x = 1, .y = 2};
twice(p);
```
`Point` は `x + x` が書けず、`Addable` を満たさない。`twice` は `Addable` を満たす型でしか候補にならず、ほかに候補もないので、`twice(p)` は呼べる関数がなく、ビルドに失敗する。
:::

今度の失敗は、本体の中ではなく、呼び出した `twice(p)` の側、すなわちオーバーロード解決の段階で起こる。候補が外れた理由も、「`Point` が `Addable` を満たさない」と、`twice` が `T` に求めるものに即して示される。テンプレートが型に何を求めるかが、本体を読まずとも、制約として宣言に現れる。

制約は、パラメータに直接書くほかに、`requires` を続けて書くこともできる。

```cpp
template <typename T>
    requires Addable<T>
T twice(T x)
{
    return x + x;
}
```

`requires Addable<T>` は、`T` が `Addable` を満たすことを求める。`template <Addable T>` と同じ制約を、別の書き方で表したものである。

標準ライブラリは、よく使う要求をコンセプトとしてあらかじめ定義している。それらは、この先の編で使う側に回ったときに見る。

:::details コンセプト以前の型の振り分け(SFINAE)
コンセプトが入る前(C++20 より前)から、テンプレートを型で振り分けるしくみはあった。テンプレートの実引数を当てはめるとき、関数の宣言の部分(引数や戻り値の型など)で置換に失敗しても、それはエラーにならず、その候補が外されるだけ、という規則がある。本章の初めに見た本体の中での失敗は、これと違ってビルドに失敗する。この規則を使い、`std::enable_if` や `<type_traits>` の道具と組み合わせて、ある型のときだけ使えるテンプレートを書いていた。この、宣言の部分での置換の失敗をエラーにしない規則を、英語では SFINAE(substitution failure is not an error)と呼ぶ。

SFINAE を使った書き方は、既存のコードや標準ライブラリ、ビルドの失敗時のメッセージで今も見かける。コンセプトは、同じ型の振り分けを、より直接に書ける。両者は同じではなく、コンセプトには、制約どうしの強さを比べて候補を絞るといった固有の働きもあるが、この編では立ち入らない。
:::
