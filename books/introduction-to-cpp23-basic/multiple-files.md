---
title: "複数ファイル"
---

宣言の章では、宣言と定義の区別が「プログラムが複数のファイルにまたがるようになってから効いてくる」と述べた。プリプロセッシングディレクティブの章では、`"name.h"` のようにダブルクォートで囲む `#include` には立ち入らなかった。本章でこれらを扱う。

ここまでは、すべてのコードを1つのファイルに書いてきた。プログラムが大きくなると、関連するまとまりごとに複数のファイルへ分けて書くことが多い。本章では、宣言の章で示したコードのうち `add` を別のファイルに分け、その際に現れる宣言と定義の区別を見る。

## `add` を別のファイルに分ける

`add` を `add.cc` という別のファイルに移し、`main` は `main.cc` に残す。

```cpp:add.cc
int add(int a, int b)
{
    return a + b;
}
```

```cpp:main.cc
#include <print>

int add(int a, int b);

int main()
{
    int result = add(2, 3);
    std::println("{}", result);
}
```

`main.cc` には、見慣れない行が1つ増えている。

```cpp
int add(int a, int b);
```

`add` の本体 `{ ... }` がなく、`;` で終わっている。これが何であるかを、次に見る。

## 宣言と定義

`main.cc` の `int add(int a, int b);` は、`add` という名前を導入し、その型が「2つの整数を受け取って整数を返す」ものであることを示す。ただし、`add` が実際にどう計算するか、その本体は与えていない。本体という実体を与えていない点で、これは宣言の章で述べた **定義** ではない。

一方、`add.cc` の `int add(int a, int b) { return a + b; }` は、本体まで与えている。こちらが `add` の定義である。

`main.cc` の側は、`add` を呼び出すために、`add` がどんな型を持つかを知っていればよい。その役割を果たすのが、本体のない宣言である。`add` が実際に何をするかは、`add.cc` の定義が受け持つ。

宣言の章で「定義は宣言の一種である」「上の3つの宣言はいずれも定義でもある」と述べた。ここで初めて、定義ではない宣言が現れた。同じ名前は複数のファイルで宣言できるが、その定義はプログラム全体で1つでなければならない。

## ビルドに両方のファイルを含める

ファイルが2つになったので、両方をビルドの対象にする。環境構築の章では、`cabin.toml` の `sources` にソースファイルを挙げた。そこに、`src/` に置いた `main.cc` と `add.cc` の両方を挙げる。

```toml
[target.hello]
type = "executable"
sources = ["src/main.cc", "src/add.cc"]
```

これで2つのファイルがどちらもビルドされる。一方だけを挙げると、もう一方は処理されないまま残り、その定義が見つからずビルドに失敗する。この失敗については、章の終わりで改めて見る。

## ヘッダ

`add` を使うファイルが増えるたびに `int add(int a, int b);` を書き写すと、書き間違いや食い違いのもとになる。そこで、この宣言を1つのファイルにまとめ、使う側がそれを取り込む形にする。宣言をまとめたこのファイルが、ヘッダである。

```cpp:add.h
int add(int a, int b);
```

```cpp:main.cc
#include <print>
#include "add.h"

int main()
{
    int result = add(2, 3);
    std::println("{}", result);
}
```

`#include "add.h"` は、`add.h` の中身、すなわち `add` の宣言を、この位置に取り込む。これで `main.cc` は `add` の宣言を直接書かずに済む。出力の章で `#include <print>` が `std::println` の宣言を取り込んだのと、働きは同じである。

## `"..."` と `<...>`

プリプロセッシングディレクティブの章で述べたとおり、`#include` には、`"add.h"` のようにダブルクォートで囲む形式と、`<print>` のように山括弧で囲む形式がある。

どちらの形式でも、取り込むファイルをどこから探すかは処理系が定める。慣習として、ダブルクォートは自分で用意したファイルに、山括弧は標準ライブラリなど処理系が提供するものに使い分けることが多い。ただしこれは慣習であって、それぞれがどこを探すかを最終的に決めるのは処理系である。

:::details 規格における2つの形式

規格は、どちらの形式についても、ファイルを探す方法を処理系定義(implementation-defined)としている。加えてダブルクォート形式には、「その方法で見つからなければ、山括弧形式として読み直す」という規定がある。したがって両者の違いは、規格上は探索方法の違いであって、「自分のファイルか、処理系が提供するものか」という区別が定められているわけではない。

:::

## 翻訳単位

`add.cc` と `main.cc` は、それぞれ個別に処理される。このソースファイル1つ分の処理の単位は **翻訳単位** と呼ばれる。2つのファイルは2つの翻訳単位として別々に処理され、最後に1つのプログラムにまとめられる。このまとめる処理は **リンク** と呼ばれる。

`main.cc` の翻訳単位は、ヘッダから取り込んだ宣言によって `add` の型を知る。`add` の実体は `add.cc` の定義から与えられ、リンクによって両者が結びつく。ファイルが複数になっても、それぞれの翻訳単位は宣言の並びである。複数ファイルとは、その宣言の並びがいくつかに分かれて書かれ、リンクによって1つのプログラムにまとめられる、ということにほかならない。

:::details ファイルの分け方は言語によって違う

C++ では(モジュールを使わない場合)、名前を使う側に宣言を取り込み、定義を別に用意し、リンクで結びつける。この宣言と定義の役割分担は、言語によっては表に出ない。

たとえば Python や Java では、ファイルを分けても、使いたい名前をモジュールやクラスの名前で取り込むだけでよく、宣言と定義を別々に書く必要はない。C++ でも、基本編では扱わないモジュールを使えば、これに近い書き方ができる。
:::

## 定義のない名前を使う

宣言は名前を導入してその型を示すだけなので、定義がどこにもなければ、その名前を呼び出してもプログラムは完成しない。

:::message
宣言はあるが定義がない
```cpp:main.cc
#include <print>

int add(int a, int b);

int main()
{
    int result = add(2, 3);
    std::println("{}", result);
}
```
`add` の宣言はあるが、その定義がどの翻訳単位にもない(`add.cc` をビルドに含めていないなど)。`add` の定義が見つからないため、ビルドに失敗する。
:::
