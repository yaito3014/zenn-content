---
title: "イテレータのコンセプト"
---

言語編のイテレータと範囲forの章で、自分の型にイテレータとセンチネルを実装した。この編でも、コンテナの標準のイテレータを使ってきた。C++20 は、このイテレータを **コンセプト** で捉え直し、C++17 までのやり方を改めた。これは、C++17 で書かれたコードを C++20 のやり方へ移すときの要でもある。

## できる操作で分かれる

イテレータは、何ができるかで分かれる。下にいくほど、できることが増える。

- 前へ進めて、一度だけ読める(input)
- 何度でも先頭からたどり直せる(forward)
- 前へも後ろへも動ける(bidirectional)
- 添字のように、離れた位置へ飛べる(random access)
- 要素がメモリ上で連続している(contiguous)

`std::vector` のイテレータは、連続まで満たす。`std::set` のイテレータは、前後に動けるが飛べない。C++20 では、これをコンセプトで確かめられる。コンセプトは、言語編のコンセプトの章で見た、型への要求である。次のそれぞれは、`true` か `false` に評価される。

```cpp
std::random_access_iterator<std::vector<int>::iterator>;   // true
std::contiguous_iterator<std::vector<int>::iterator>;      // true
std::bidirectional_iterator<std::set<int>::iterator>;      // true
std::random_access_iterator<std::set<int>::iterator>;      // false
```

`std::vector` のイテレータは飛べる(random access)が、`std::set` のイテレータは前後に動くだけで飛べない。`std::set` は要素を一つずつ別の場所に置いて保つ(連想コンテナの章)ので、添字のように離れた位置へ一度に飛べないからである。飛ぼうとすると、ビルドの時点で分かる。

:::message
飛べないイテレータで飛ぶ
```cpp
std::set<int> s = {1, 2, 3};
auto it = s.begin();
it += 2;
```
`std::set` のイテレータは前後に動くだけで、`+= 2` のように離れた位置へは飛べない。ビルドに失敗する。`std::vector` のイテレータなら、`+= 2` で飛べる。
:::

## タグからコンセプトへ

C++17 では、イテレータの種類を **タグ** で表した。`std::iterator_traits` を通して `iterator_category` というタグ(`std::random_access_iterator_tag` など)を見て、種類を見分けた。C++20 では、種類をコンセプト(`std::random_access_iterator` など)で表す。`std::random_access_iterator<T>` は、`T` が飛べるイテレータの要求を満たすかを、その場で確かめる。タグを別に宣言しておかなくても、できる操作がそろっていれば、そのコンセプトを満たす。

:::details コンセプト化は、タグの言い換えではない(proxy reference)
コンセプトへの移行は、タグに別名を付けただけではない。要求そのものが定義し直された。その代表が、`*it` で取り出す型である。

C++17 では、前方(forward)以上のイテレータは、`*it` が本物の参照(`T&`)を返すことを求めた。このため、`*it` が参照のように振る舞う代理のオブジェクト(英語では proxy reference と呼ぶ)を返すイテレータは、前方イテレータになれなかった。たとえば `std::vector<bool>` は、要素を詰めて持つので、`*it` は `bool&` ではなく代理のオブジェクトを返す。

C++20 のコンセプトは、`*it` の型を値の型と関係づける形で定義し直され、こうした代理を返すイテレータも、できる操作がそろっていれば満たせるようになった。代理をやり取りするための仕組み(`std::ranges::iter_move` など)もあるが、この編では立ち入らない。
:::

## センチネル

言語編で作った範囲では、`begin` がイテレータを、`end` がセンチネルを返し、両者は別の型でよかった。`end` が `begin` と別の型でよくなったのは、C++17 からである。C++20 では、この終端の役割が `std::sentinel_for` というコンセプトとして正式に規定され、`std::ranges` 全体で統一的に扱えるようになった。別の型のセンチネルが許されることで、終端が「ある条件を満たしたところ」で決まる範囲も、自然に表せる。たとえば、ヌル文字までの範囲を、ヌル文字に出会ったかで終端を判定するセンチネルで表せる。

## C++17 から C++20 へ

イテレータの捉え方は、まとめると次のように変わった。

- 種類は、タグ(`iterator_category`)から、コンセプト(`std::forward_iterator` など)へ。
- 終端は、C++17 で `begin` と別の型の `end` が使えるようになり、C++20 で `std::sentinel_for` コンセプトとして規定された。
- アルゴリズムは、イテレータの組 `(first, last)` を渡すことから、範囲を直接渡すこと(`std::ranges`)へ。

C++17 のイテレータは C++20 でもおおむねそのまま使える。新しく書くなら、タグを定義する代わりに、満たしたいコンセプトの要求をそろえ、イテレータの組の代わりに範囲を渡す。

ここまで、コンテナのイテレータを、その能力で捉えるコンセプトとして見てきた。
